# DEV PROTOCOL — Как мы строим «Архив Пустоты» (Godot 4)
**Роль файла:** операционный мануал. Дизайнер (я) выдаёт промты-слои. Ты раскидываешь их по трём агентам Claude. Этот файл объясняет Godot-архитектуру, конвенции и — главное — как **не забивать контекст**.

> Движок: **Godot 4.x (рекомендуется 4.3+), GDScript.** Агент должен сверять API под установленную версию. Запуск — клавиша **F5** в редакторе. Ноль импортированных ассетов (только `.gd`, `.gdshader`, минимальные `.tscn`, `project.godot`, `.md`).

## 1. Три агента (роутинг)

Каждый промт слоя помечен тегом маршрута в шапке. Ориентир:

| Агент | Имя | Зона ответственности | Что трогает |
|---|---|---|---|
| **A1** | **«ЯДРО»** | Глобальные механики, архитектура, новые системы, большие развёрнутые слои | `scripts/core/`, `scripts/systems/`, автозагрузки, шейдеры |
| **A2** | **«СБОРКА»** | Контент и средние задачи: новые здания/исследования/аномалии (как ДАННЫЕ), новые UI-панели, балансные проходы | `scripts/data/`, `scripts/ui/panels/`, Theme-твики |
| **A3** | **«НАЛАДКА»** | Мелкие правки, фикс багов, полировка, твики шейдеров/Theme, значения | точечно везде, маленькие диффы |

**Почему это работает:** контент — data-driven (см. §3): логику пишет A1 один раз, дальше A2 добавляет контент как записи данных без переписывания систем. A3 чистит. Контексты агентов почти не пересекаются.

Правило роутинга: «новая система или меняет, как игра работает» → A1. «Добавить контент в существующую систему / новую панель» → A2. «Что-то сломалось / подвинуть/покрасить» → A3.

## 2. Стратегия памяти (решение главной прошлой проблемы)

1. **`PROJECT_STATE.md` — живой снимок билда.** Агент обновляет его в конце КАЖДОГО слоя. Новый чат онбордится за одно чтение, а не пересказом истории. Шаблон в §6.
2. **Каждый промт самодостаточен.** Агенту не нужна история чата: вся конкретика — в промте слоя. Историю можно чистить между слоями.
3. **Ссылайся, не вставляй.** Файлы указываются путём (`scripts/core/game_state.gd`), а не вставляются целиком, если не крошечные.
4. **Граница «НЕ делать».** В каждом промте есть out-of-scope — чтобы агент не строил весь эндгейм и не забивал контекст (ровно то, что убило прошлый заход).
5. **Data-driven код** = маленькие локальные диффы вместо переписывания.
6. **`CHANGELOG.md`** — одна строка на слой. Никогда не раздувать.

## 3. Архитектура проекта (Godot-нативная)

Концептуальные слои те же (ядро / данные / системы / UI), но на средствах Godot:

- **ЯДРО → автозагрузки (singletons).** Глобальное состояние, шина событий, игровой цикл, сохранение — это `AutoLoad`-узлы. Регистрируются в `project.godot` (секция `[autoload]`). Доступны по имени из любого скрипта.
- **СОБЫТИЯ → встроенные сигналы.** Свой `Events`-автозагрузка объявляет сигналы игры. Связь систем и UI — только через них (никаких прямых ссылок между узлами через всё дерево).
- **ДАННЫЕ → статические классы** (`class_name XxxDB` со `static func`/`const`). Контент описывается как данные. Зона A2. (Позже при желании можно мигрировать на кастомные `Resource`, но по умолчанию — данные в GDScript: меньше файлов, AI-дружелюбнее, чище диффы.)
- **СИСТЕМЫ → статические классы** (`class_name Production` и т.п.), читают `GameState` + данные, мутируют состояние. Никакого UI внутри. Зона A1.
- **UI → строится кодом.** Панели = `class_name XPanel extends <Container>`, которые **сами собирают своё дерево узлов в `_ready()`** и обновляют свои `Label`-ы по сигналам `Events`. Никаких хрупких `.tscn` для каждой панели, никакого централизованного «render». Единственная сцена — минимальный `main.tscn` (корневой `Control` + `main.gd`), всё остальное — в коде.
- **ВИЗУАЛ → шейдеры (`.gdshader`) и Theme.** CRT-оверлей и «ядро» — `ColorRect` с `ShaderMaterial`. Стилизация Control-ов — `Theme`, который **строится в коде** (`ThemeBuilder.build()`), с моноширинным `SystemFont`. Ноль картинок/шрифтов-файлов.

Целевое дерево (полное; каждый слой создаёт только нужное ему):

```
void-archive/
├─ project.godot
├─ README.md
├─ PROJECT_STATE.md      ← живой снимок (агенты обновляют)
├─ CHANGELOG.md
├─ scenes/
│  └─ main.tscn          ← минимальный: Control-корень + main.gd
├─ shaders/

│  └─ core.gdshader      ← светящееся вращающееся «ядро»
```

### Контракты модулей (минимум, чтобы слои стыковались)
- **GameState** (autoload) → `resources`/`buildings`/`meta` (Dictionary) + `get_resource(id)`, `add_resource(id, amount)` (эмитит `Events.resource_changed`), `reset_to_default()`.
- **Events** (autoload) → сигналы: `resource_changed(id, value)`, `data_gained(amount)`, `log_message(text, type)`, `tick(delta)`, `game_loaded()`.
- **GameLoop** (autoload) → `_process(delta)`: кламп `dt`, вызвать `Production.update(dt)`, эмитнуть `tick`, автосейв по таймеру. Хранит `current_data_rate`.
- **SaveManager** (autoload) → `save_game()`, `load_game()`, `wipe()`; оффлайн-прогресс на загрузке (потолок по времени).
- **Format** (static) → `num(n) -> String`, `rate(n) -> String`.
- **Palette** (static) → цветовые константы (`const AMBER := Color("ffb000")` и т.д.).
- **ThemeBuilder** (static) → `build() -> Theme` (моноширинный `SystemFont`, цвета, `StyleBoxFlat`).
- **Production** (static) → `update(delta)`, `compute_data_rate() -> float`.
- **UI-панели** (`class_name … extends Container`) → строят дерево в `_ready()`, подключаются к `Events`, обновляют свои `Label`-ы. Без централизованного render.

## 4. Конвенции кода (GDScript)
- Один источник истины — автозагрузка `GameState`. Никаких разрозненных глобалок.
- Системы — «чистые по входу»: вход = `GameState` + данные, выход = мутация состояния. Никаких узлов/UI внутри систем.
- Связь систем и UI — только через сигналы `Events`. Узлы не лезут друг к другу через всё дерево.
- Контент — всегда данные в `scripts/data/`. Добавить здание = добавить запись, не править систему.
- Числа форматируются ТОЛЬКО через `Format`.
- Статическая типизация по возможности (`func f(x: float) -> String:`).
- Именование: `snake_case` для функций/переменных/файлов; `PascalCase` для `class_name` и имён автозагрузок; сигналы — `snake_case`.
- Тексты UI игры — русские; системные «логи» — стилизованный смешанный терминальный язык (см. GDD §8).
- Комментарии короткие и по делу.

```
ПРОЧИТАЙ СНАЧАЛА: PROJECT_STATE.md + <только релевантные файлы по путям>.

SCOPE (создать/изменить только это): <список файлов>.
```

## 8. Definition of Done (для любого слоя)
- Открывается и запускается в Godot 4.x (F5) — **ноль ошибок** в Output/Debugger.
- Все пункты «Критериев приёмки» слоя выполнены и проверяемы вручную.
- В проекте **нет импортированных медиа-ассетов** (только `.gd`, `.gdshader`, `.tscn`, `project.godot`, `.md`).
- `PROJECT_STATE.md` и `CHANGELOG.md` обновлены.
- Ничего из OUT OF SCOPE не реализовано (скоуп держим).
